## 2024-05-24 - Missing Input Validation in Invitation API | **Learning:** API routes accepting POST/PUT requests for creating/updating invitations were directly reading from request bodies, exposing the application to XSS vulnerabilities via un-sanitized fields like title and description. | **Prevention:** Always pipe incoming request payloads through established validation functions like `validateInvitationData` and strictly use `validation.sanitizedData` before persisting to the database.
## 2026-03-08 - Missing Authentication on Template API Routes | **Learning:** The `POST`, `PUT`, and `DELETE` methods on `/api/templates` and `/api/templates/[id]` were missing `getServerSession` checks, despite comments indicating they were admin-only. This allowed any unauthenticated user to modify or delete default templates, bypassing intended authorization. | **Prevention:** Always ensure that any API route performing state-mutating operations (POST, PUT, DELETE) explicitly checks for a valid session using `getServerSession` before proceeding with database operations, especially when using the privileged `supabaseAdmin` client.
## 2026-03-12 - Insecure JSON-LD Injection in Script Tags | **Learning:** Using `JSON.stringify` directly within `dangerouslySetInnerHTML` for `<script type="application/ld+json">` is vulnerable to XSS if user-controlled strings contain `</script>`, as the browser terminates the script block regardless of string delimiters. | **Prevention:** Use a dedicated serialization utility like `serializeJsonLd` that escapes `<`, `>`, and `&` to Unicode sequences (`\u003c`, `\u003e`, `\u0026`) before injecting into HTML.
## 2026-03-17 - Secure Data Access for Guest RSVPs | **Learning:** Writing guest RSVPs directly to Supabase via the anonymous client from `src/app/api/rsvp` violates strict API architecture rules and will fail due to write RLS. | **Prevention:** All data access bypassing write RLS must securely leverage the service role (`supabaseAdmin`) via `src/lib/database-supabase.ts` and utilize strict backend validation (e.g. `validateRSVPData`) to enforce schema rules.
## 2026-03-30 - Unauthenticated External API Calls in Next.js Routes | **Learning:** Always check Next.js App Router API endpoints that call external services (like Gemini) to ensure they are protected by `getServerSession`, even if part of the route is meant for unauthenticated demo access. | **Prevention:** Manually verify `getServerSession` at the top of any server-side logic that interacts with external APIs or databases (unless explicitly public like `/api/rsvp`), before the sensitive operation occurs.
## 2026-04-06 - Missing share_token validation in RSVP endpoint | **Learning:** Guest access relies on `share_token` to bypass session checks safely. However, the `POST /api/rsvp` endpoint previously only validated the `invitation_id` provided in the body, which allowed an IDOR (Insecure Direct Object Reference) attack where anyone with an `invitation_id` could spam RSVPs. | **Prevention:** To prevent IDOR vulnerabilities in link-based guest access routes, validate operations using the secret `share_token` alongside the public identifier (e.g., `invitation_id`). Ensure both frontend hooks and tests are updated to provide this token.
